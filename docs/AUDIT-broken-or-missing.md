# FireISP 5.0 — Audit: What's Broken or Missing
_Generated by an automated 13-dimension static audit (database → backend → web GUI). Findings were adversarially verified; refuted ones removed._

## Executive summary

**Total verified findings: 60** (after deduplication) — **5 critical · 25 high · 21 medium · 9 low** (the 9 low includes several explicit "this is healthy / out of scope" confirmations).

Overall, FireISP 5.0 is a large, mostly-functional platform whose core billing, RADIUS, and client flows are backed by real implementations and a broad test suite. The genuinely **BROKEN** items cluster in three places: (1) a family of **ORM-model/schema column mismatches** in newer security/data-protection tables (AdminIpAllowlist, ApiKeyRateLimit, BlackholeRoute, DataMaskingRule, DnsBlocklist, EmailLog, EncryptionKeyMetadata, PasswordPolicy) whose CRUD endpoints will throw "Unknown column" SQL errors on every create/update; (2) **scheduled-task dispatch gaps** where seeded task names (six FTTH tasks, SNMP discovery/trap tasks) have no matching `case` in `taskRunner.js`, so those monitors silently never run; and (3) **frontend↔backend API path mismatches** that 404 entire pages (the Topology Map page double-prefixes `/v1`, Inventory calls a hyphenated path, the DND tab uses the wrong param name, and the Subscriber Certificate page is never routed). The most financially dangerous BROKEN item is the payment gateway: unrecognized providers are marked `succeeded` with a fake reference and no API call, and `refund()` never calls the processor.

The larger category is **MISSING/INCOMPLETE** rather than broken: third-party integrations, the scripting/automation executor, non-MikroTik router drivers, and several metrics are honestly-labeled stubs; rollbacks are absent for migrations 001–129; and roughly **half the frontend pages (71/135) bypass i18n** despite complete Spanish/Portuguese translations existing. The single most important theme: **the database schema and the code that talks to it have drifted apart** — both at the ORM layer (column-name mismatches) and at the orchestration layer (task-name mismatches) — so features that look complete on paper fail at runtime.

> Note on multi-tenancy: this is a single-tenant-per-ISP deployment, so cross-org data-isolation gaps were intentionally NOT flagged. The invoice/quote/credit-note unique-key findings remain because they cause **duplicate-key INSERT failures**, not because of cross-tenant leakage.

## Top priorities (do these first)

1. **[critical] Payment gateway fakes success & never refunds** — generic/unrecognized providers are marked `succeeded` with a `Date.now()` reference and no API call (`src/services/paymentGatewayService.js:107-125`); `refund()` only flips DB status to `refunded` without calling Stripe/Conekta (`:265-278`). Phantom transactions + customers never actually refunded.
2. **[critical] Topology Map page 404s entirely** — all calls use `/v1/topology/*` against a client whose baseUrl is already `/api/v1`, yielding `/api/v1/v1/topology/...` (`frontend/src/pages/TopologyMapPage.tsx:133,250,258,266,377,400`).
3. **[critical] Invoice / quote / credit-note number UNIQUE keys omit `organization_id`** — `database/schema.sql:725, 951, 3112`; correct pattern exists at `service_orders` (`:231`). Concurrent cross-org number generation throws duplicate-key on INSERT, breaking billing/CFDI audit trails.
4. **[high] FTTH OLT/ONU monitoring never executes** — six tasks seeded in `database/migrations/269_seed_ftth_scheduled_tasks.sql` have no `case` in `src/services/taskRunner.js` (switch at lines 52-159); dispatch falls through to "Unknown task".
5. **[high] SNMP discovery & trap tasks never run** — seeded `snmp_discovery_poll` / `snmp_trap_receiver` (`migrations/254:23,45`) don't match the `snmp_poll` / `snmp_trap_receiver_restart` cases in `taskRunner.js:62,64`. No metrics collected, trap listener never starts.
6. **[high] Eight ORM models reference non-existent columns** — CRUD INSERT/UPDATE will throw "Unknown column": AdminIpAllowlist (`ip_address`/`expires_at` vs `cidr`), ApiKeyRateLimit (`token_id` vs `api_token_id`), BlackholeRoute, DataMaskingRule, DnsBlocklist, EmailLog, EncryptionKeyMetadata, PasswordPolicy. Breaks admin IP allowlist, DDoS blackholing, data masking, encryption-key mgmt, password policy.
7. **[high] Mass-assignment via dynamic column interpolation** — `src/routes/trafficEngineering.js:70,92` build INSERT/UPDATE column lists from raw `Object.keys(req.body)` with no whitelist; `validate` middleware doesn't strip extra fields, so `organization_id`/`deleted_at` are settable on updates.
8. **[high] Automation scripting & batch ops fully stubbed** — `scriptingService.executeScript()` only records a `queued` row (no child_process); `automationService.applyBatchOperation()` and `executeProvisioningStage('configure_device')` log stubs (`src/services/automationService.js:7-10,221-224,307-310`). Bulk plan/suspend/config workflows never execute.
9. **[high] Inventory & DND tabs 404 / fail to load** — frontend calls `/inventory-items` but backend is `/inventory/items` (`InventoryManagement.tsx:104` vs `src/routes/inventory.js:22`); DND tab uses `{id}` but backend/schema expect `{clientId}` (`ClientCommunicationPrefs.tsx:57`).
10. **[high] Subscriber Certificate UI is unreachable** — `SubscriberCertificateList.tsx` and its backend route (`/subscriber-certificates`, mounted `app.js:568`) exist, but the page is never imported or routed in `frontend/src/App.tsx`. Plus **71/135 frontend pages bypass i18n** entirely despite complete es/pt-BR translations.

## What is actually working / out of scope

- **Schema integrity is solid:** all 358 migrations map cleanly to `schema.sql` (322 CREATE TABLE, no drift); all INSERT migrations use `WHERE NOT EXISTS`/`INSERT IGNORE` guards; all ALTER-ADD-COLUMN use `INFORMATION_SCHEMA` guards; no PostgreSQL syntax; no circular/orphaned FKs. The DB foundation is healthy — the problems are at the code↔schema boundary, not in the schema itself.
- **Core services are real and tested:** billing, auth/login, CFDI cancel/amend (with SAT rejection handling), suspension logic, and portal auth/service-requests all have substantive service-level test coverage (242 backend test files).
- **i18n infrastructure is complete:** Spanish and Portuguese are fully and professionally translated (2768 keys each); the only gap is wiring them into ~71 pages — a high-ROI, low-risk follow-up.
- **MikroTik path is live:** router-driver dispatch works for MikroTik; the gaps are other vendors (explicitly stubbed) and a handful of integration adapters.
- **Single-tenant deployment note:** this is a single-tenant-per-ISP install, so cross-org **data-isolation** gaps were intentionally NOT flagged. The retained `organization_id` unique-key findings are about **duplicate-key INSERT failures during number generation**, not cross-tenant data leakage.

## Methodology & limitations

- **Static analysis only:** no live database or running app server; findings rest on reading source, schema, migrations, OpenAPI spec, and grep/git evidence (file:line quoted throughout).
- **Adversarial verification:** every finding was independently re-checked; refuted ones were removed and several severities were adjusted up or down (e.g., the "SQL injection" claims were downgraded to mass-assignment / defense-in-depth once parameterization was confirmed).
- **Confidence varies by tier:** critical/high findings were re-verified with concrete evidence; many medium/low findings were passed through from their original dimension without an independent second pass and should be spot-checked before action.
- **A runtime audit would add:** actual SQL execution against the schema (confirming the "Unknown column" failures and unique-key collisions), scheduler runs proving the FTTH/SNMP tasks no-op, live HTTP probes of the 404'ing frontend pages, and end-to-end payment/refund and suspension flows that static analysis can only infer.

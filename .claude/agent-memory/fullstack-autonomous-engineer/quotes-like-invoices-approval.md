Built on branch `feat/quotes-like-invoices-approval`, in two passes (a mid-task correction changed the create flow — see below).

**Pass 1 — QuoteDetail (kept as-is, still current):** `frontend/src/pages/QuoteDetail.tsx` (new) mirrors `InvoiceDetail.tsx`'s structure — header, line-items table, Approve/Reject/Convert-to-Invoice actions, PDF download, an `EditQuoteModal` (metadata + manual totals override). Plus `POST /quotes/:id/approve` / `POST /quotes/:id/reject` in `src/routes/quotes.js` (both `quotes.update`, no new permission — user-confirmed decision), and a `POST /quotes/:id/convert-to-invoice` gate: `if (quote.status !== 'accepted') return 409 {code:'QUOTE_NOT_ACCEPTED'}`, placed after the existing 404 check, before the transaction. Approve/reject are deliberately lenient — no status-transition guard — via `Quote.update(id, {status}, orgId)` (org-scoped + soft-delete-guarded for free via BaseModel). `quotes` has no `invoice_id` back-reference, so "already converted" isn't detectable without a migration — left undetected, noted in the route comment. **Key finding that shaped Pass 2:** `InvoiceDetail.tsx` has NO "add item" UI at all — invoices are built in one shot via `GenerateInvoiceModal`, never per-item. QuoteDetail's own per-item "Add Item" form (POST item → refetch → `computeTotals` → PUT quote) is real and stays — it's how you add MORE items to an already-created quote — but it turned out not to be how quotes should be *created*.

**Pass 2 — the coordinator corrected the creation flow mid-task:** the user's actual ask was "make a new quote like making an invoice," and the real invoice-creation UX is `GenerateInvoiceModal.tsx` (pick client → add contract/product/custom line items → submit ALL at once to `POST /invoices/generate`), not a create-draft-then-add-items-one-at-a-time flow. This added:

- **`POST /quotes/generate`** in `src/routes/quotes.js`, mirroring `POST /invoices/generate`'s FLEXIBLE format as closely as correctness allows. One deliberate, necessary deviation: it does **NOT** call `billingService.generateBillingPeriod()` for `type:'contract'` items — that function has side effects (creates/reuses a REAL `billing_periods` row, later marked `'invoiced'`). A quote is just an estimate that may never be accepted, so contract-item price resolution here is read-only (contract + plan lookup, `contract.price_override || plan.price`, no `billing_periods` write). `'product'`/`'custom'` items are handled identically to the invoice route (the type tag is a frontend-only distinction — both trust client-supplied description/quantity/unit_price; the catalog lookup that fills those in is entirely client-side, same `GET /plans/addons/catalog` `GenerateInvoiceModal` already uses). Tax math is the exact same fraction formula (`subtotal * taxPct`, `Math.round(x*100)/100`, org's default `tax_rates` row) — verified with a dedicated no-100x-bug regression test.
- **Migration 389** (`database/migrations/389_organization_quote_sequences.sql` + rollback): `organization_quote_sequences` mirrors `organization_invoice_sequences` (migration 381) shape-for-shape — same `INSERT IGNORE` + `UPDATE ... LAST_INSERT_ID(next_number)+1` idiom (see [[mysql-atomic-sequence-idiom]] — a single `ON DUPLICATE KEY UPDATE` doesn't reliably run its `LAST_INSERT_ID()` on a fresh insert into a table with no AUTO_INCREMENT column), seeded from existing `quotes.quote_number` (`QUO-%` parse + `GREATEST()` with `COUNT(*)`, same as 381). New `billingService.nextQuoteNumber(conn, orgId)` (mirrors `nextInvoiceNumber` exactly, `QUO-` prefix) — dedicated unit tests added in `tests/billingService.test.js` right after the `nextInvoiceNumber` describe block (5 tests: first-call, increment, sentinel-0, beyond-999999, never-queries-quotes-table).
- **Plain `POST /quotes`** (the crudController-backed create route) now auto-assigns `quote_number` via `nextQuoteNumber` in its own short-lived transaction when the caller omits it — explicit values are still honored. This **superseded and removed** the earlier "quote_number required" frontend workaround entirely (see below) — it's optional again, exactly like `invoice_number`.
- **`frontend/src/components/GenerateQuoteModal.tsx`** (new) — a close clone of `GenerateInvoiceModal.tsx` (same client/contract/product-catalog `useQuery` hooks, same three item-type buttons, same validation), submitting to `POST /quotes/generate`. `onGenerated(quote)` receives the created quote so the caller can navigate — unlike `GenerateInvoiceModal`'s bare `onGenerated: () => void`, since `QuoteList` needs the new id to route to `/quotes/:id`.
- **`QuoteList.tsx`'s "New Quote" now opens `GenerateQuoteModal`**, replacing the short-lived `CreateQuoteModal` (client + quote-number + valid-until + tax-rate + notes) that Pass 1 built specifically to route around the then-missing auto-numbering. `CreateQuoteModal` and its required-quote_number workaround were deleted outright — do not resurrect them; `GenerateQuoteModal` is now the only create path.

**Test gotcha found in Pass 2, worth remembering broadly:** `fireEvent.change` on a `<select>` to a value whose `<option>` doesn't exist YET (e.g. the options are populated by an async `useQuery` that hasn't resolved) is silently dropped — the DOM's `value` stays at whatever the currently-rendered options allow (usually the empty placeholder), and the component's `onChange` fires with the OLD/empty value, not the one you asked for. Always `await screen.findByRole('option', {name: ...})` (or otherwise wait for the query to resolve) before `fireEvent.change`-ing a select whose options come from an async fetch. Cost real debugging time here — `mockApiPost` silently never got called, with the mutation's own guard silently short-circuiting on an empty `clientId`.

**Test gotcha carried over from Pass 1:** `getByLabelText` needs the `<label>` to *wrap* the input (implicit association) or `htmlFor`/`id` (explicit) — `GenerateInvoiceModal`/`GenerateQuoteModal`'s Client `<label>` is a SIBLING of its `<select>` with no association at all (a real, pre-existing a11y gap, faithfully cloned since "clone it" was the explicit instruction). Don't use `getByLabelText` against it in tests — find the select via its placeholder option instead (`screen.getByText('— select client —').closest('select')`), matching how `GenerateInvoiceModal.test.tsx` avoids the same trap entirely by using `findByRole('option', ...)`/`getByTitle(...)` instead of label queries.

## Sanitize middleware removed (migration-free, PR fix/sanitize-output-encoding-radius-credentials)

`src/middleware/sanitize.js` (global `app.use(sanitize)` at former app.js:298, HTML-entity-encoded every string in every `req.body`) is **deleted**, not just disabled. It corrupted two things: (1) JSON-stringified fields going into JSON-typed columns (quotes became `&quot;`, MySQL's JSON validator rejected it â†’ unmapped `ER_INVALID_JSON_TEXT`/errno 3140 â†’ raw 500, e.g. IFT statistical reports), (2) free-text columns permanently stored `&#x27;`/`&amp;` instead of `'`/`&` (client names, CFDI `receptor_nombre`, KB articles).

**New security posture** (documented as a comment at the former mount point in `src/app.js`): output-side encoding only â€” React/JSX auto-escaping everywhere, `DOMPurify.sanitize()` at the one `dangerouslySetInnerHTML` sink (`frontend/src/pages/portal/PortalKb.tsx`), `cfdiService.escapeXml()` for CFDI XML, and a small local `escapeHtmlForTemplate()` helper in `notificationService.js` that HTML-escapes interpolated template variables **only for the `email` channel** (not sms/whatsapp â€” those are plain text, escaping would corrupt what the subscriber reads).

`app.js`'s error handler gained an `ER_INVALID_JSON_TEXT`/errno 3140 â†’ 422 `INVALID_JSON` mapping as defense-in-depth (alongside the existing `ER_SIGNAL_EXCEPTION`/`ER_DUP_ENTRY`/`ER_NO_REFERENCED_ROW_2` mappings) â€” not expected to be hit by normal traffic anymore, but guards any future caller.

**Data repair**: `src/scripts/repair-entity-encoded-fields.js` (repo convention is `src/scripts/`, NOT a root-level `scripts/` dir â€” the original fix spec assumed the wrong path). Dry-run by default, `--apply` to write. Decodes the 5 known entities (`&amp;` last, to avoid mis-collapsing a genuine `&amp;amp;`). Curated column list (`clients`, `leads`, `tickets`, `portal_kb_articles`, `message_templates` free-text columns) â€” NOT an exhaustive schema sweep. `cfdi_documents.receptor_nombre` is handled separately: only `uuid IS NULL` (draft/unstamped) rows are auto-repaired; stamped rows are printed for manual finance/compliance review, never written (a stamped CFDI's legal name was already filed with the SAT). This is a standalone script â€” deliberately NOT registered in `taskRunner.js`, must run manually, and only after the sanitize.js removal has deployed (running it before means new legitimate data could get mistaken for corruption).

`tests/sanitize.test.js` was deleted (the middleware it tested no longer exists). Regression coverage lives in `tests/routeIntegration.test.js` (IFT JSON round-trip via `POST /api/v1/ift-statistical-reports`, client-name-with-apostrophe round-trip via `POST /api/clients`) and `frontend/src/pages/portal/__tests__/PortalKb.test.tsx` (DOMPurify strips an `<img onerror>` payload while preserving `<b>` formatting â€” confirmed DOMPurify's default config strips ALL `on*` attributes regardless of tag, without needing custom config).

## radius.credentials.view permission (migration 383)

Fixes: `GET /radius`, `GET /radius/:id`, `GET /radius/contract/:contractId` used to return the cleartext PPPoE `password` column gated only by `devices.view` â€” and since migration 119 grants `readonly` every `*.view` permission by wildcard, a pure view-only/auditor role got cleartext subscriber passwords.

**Design** (differs from the original fix spec's draft in a few places â€” spec assumed granting `support`/`noc_operator` `radius.credentials.view` would be a no-op since they lack `devices.view`; the actual chosen fix instead makes the two new `/credentials` routes standalone, gated ONLY by `radius.credentials.view`, so a role can get credentials without gaining broader device access â€” see [[network-authz-hardening]] for the sibling cross-tenant-leak pattern this follows):

- `Radius.SAFE_COLUMNS` (all columns except `password`) used by the base `findByContract(contractId, orgId)`. New `findByContractCredentials(contractId, orgId)` and `findCredentialsById(id, orgId)` return the full row (incl. password) and are only called from the new `radius.credentials.view`-gated routes.
- `src/utils/radiusSanitize.js` mirrors `src/utils/userSanitize.js`'s idiom exactly: a plain function (`sanitizeRadius`) in its own module, NOT just a model static, "so that test suites which auto-mock the model do not replace it with an undefined-returning jest.fn()". `crudController(Radius, { serialize: Radius.sanitize })` wires it into list/get. **Reusable pattern**: any future sensitive-column strip on a model should follow this same `src/utils/<model>Sanitize.js` + `Model.sanitize` delegating-static shape.
- Cross-tenant leak fix folded into the same migration/PR (found by both the fix spec and independently expected per CLAUDE.md's SWEEP_FOLLOWUP precedent): `radius` has no `organization_id` column (`hasOrgScope = false`) and neither `findByContract` nor the new credentials routes verified org ownership before this PR. Fixed via `JOIN contracts c ON c.id = r.contract_id AND c.organization_id = ?` for the two contract-scoped queries, and `JOIN clients cl ON cl.id = r.client_id AND cl.organization_id = ?` for the id-scoped `findCredentialsById` â€” deliberately joins through `clients` (NOT `contracts`) because `radius.contract_id` is nullable (`ON DELETE SET NULL` when its contract is deleted) while `client_id` is `NOT NULL`/`ON DELETE RESTRICT`; joining through clients means an orphaned-contract radius row is still resolvable by its rightful org instead of permanently 404ing for legitimate staff. **This is a deliberate deviation from the fix spec's literal "join through contracts" suggestion for the id-scoped route** â€” the spec explicitly left the NULL-contract-row decision open ("decide by reading the schema/usage â€” do not leak cross-org rows"), and the client-join approach satisfies the non-leak invariant while being more robust for that edge case.
- Grant matrix (migration 383, DML-only, `INSERT IGNORE`): `admin`, `super_admin`, `technician`, `support`, `noc_operator` get the new slug; `manager` too but it's a documented no-op (no `roles` row named 'manager' exists â€” same pre-existing gap noted in [[network-authz-hardening]] area). `billing`, `readonly`, `auditor`, `reseller_admin` excluded, matching migration 377's credential-module carve-out convention.
- Frontend split-fetch UX: `ContractDetail.tsx`'s `PppoeTab` and `ContractList.tsx`'s `ContractDetailModal`/`RadiusAccountCard` both now do TWO queries â€” the base (password-free) fetch always runs first, and a second `/credentials`-suffixed fetch is `enabled` only once an account is known to exist; on 403 from the credentials fetch, the UI shows the account (username/status/etc.) with the password field replaced by an "insufficient permission" note instead of hiding the whole thing. `ContractDetail.tsx` has NO i18next integration at all (deliberately kept the new copy as plain hardcoded English there, matching the file's existing convention); `ContractList.tsx` DOES use `useTranslation` â€” new copy went through `t('contractList.passwordForbidden')` with en/es/pt-BR keys.

## Follow-up fix: base routes accept EITHER permission (2nd commit, same PR)

First-pass implementation left a real gap: the base routes stayed gated by `devices.view` **only**, and `support`/`super_admin`/`noc_operator` have never held `devices.view` (confirmed by grepping every migration â€” only `technician` explicit, `readonly` via 119's wildcard, and org-membership `admin` via 119's `ON TRUE` blanket grant hold it). Since the frontend's split-fetch depends on the base fetch succeeding *before* the credentials fetch is even attempted (`enabled: !!account`), those three roles could hit the `/credentials` routes directly via API but never actually saw a password in the product UI â€” the base fetch 403'd first and the whole PPPoE tab/modal died. This broke the user's hard constraint (support must see credentials end-to-end) despite the backend nominally granting it.

Fix (coordinator-directed, after verifying `src/middleware/rbac.js`'s `requirePermission(...permissions)` really is OR semantics â€” `requiredPermissions.some(p => permissions.includes(p))`, confirmed by reading the source, not assumed): the three base routes (`GET /radius/contract/:contractId`, and the two `crudController(Radius)` list/get lines for `GET /radius` and `GET /radius/:id`) now pass `requirePermission('devices.view', 'radius.credentials.view')`. Zero `role_permissions` changes â€” this only widens which permission the *route guard* accepts; responses stay password-free either way since serialization (`Radius.sanitize` / explicit `SAFE_COLUMNS` projection) is unconditional and independent of which permission unlocked the request. **General lesson for future permission-scoping PRs in this repo: when you add a narrower permission that's supposed to let a role reach a "base" view without a broader coarse permission, check whether any OTHER route in the same flow still gates on the coarse permission exclusively â€” a UI that chains two fetches (base â†’ gated extra) breaks silently for the exact role the narrow permission was added for, unless the base fetch's gate is widened too.**

## Known adjacent gap (flagged, not fixed â€” explicitly out of scope)

`POST /contracts/:id/regenerate-pppoe` (gated by `contracts.update`, a legitimate write permission) returns the freshly-rotated password directly in its response, regardless of whether the caller holds `radius.credentials.view`. A role with `contracts.update` but not `radius.credentials.view` can therefore still see a PPPoE password by triggering a rotation. This is the same adjacent surface the original fix spec flagged as "worth confirming is acceptably scoped" â€” not fixed here since the brief explicitly scoped this PR to the three GET-credential routes and the cross-tenant leak on those specific queries.
